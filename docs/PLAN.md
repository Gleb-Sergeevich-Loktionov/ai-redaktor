## Definition of Done (приёмка V1)
- Проекты: создание/выбор/обновление «библии»; данные в БД переживают рестарт.
- Редактура учитывает «библию» активного проекта; возвращает исправленный текст + саммари + перечень правок.
- Неоправданные англицизмы заменены в тексте; перечень показывает тип/фрагмент/правку/причину.
- Клиентские правки: поток сводит версии и комментирует.
- Понятные сообщения на неподдерживаемый/повреждённый/пустой ввод и превышение лимита.
- Один деплой; БД на volume; словарь/правила правятся без изменения кода.

## Риски и митигации

| Риск | Митигация |
|---|---|
| Состояние/БД усложняет V0 | SQLite (один файл) + репозиторий-шов; хранится только `bible_text`, истории нет; бэкап файла БД. |
| Перечень правок длиннее лимита Telegram (4096) | Длинный перечень отдаём отдельным файлом; в сообщении — саммари. |
| Неоднозначность «клиентских правок» | Зафиксировать UX (двухшаговый диалог) и формат результата (сведённый текст + перечень); итерация на реальных примерах. |
| Нет ACL на «библию» проекта | Внутренний инструмент; правит участник чата. Полноценный ACL — V2. |
| Потеря вёрстки в `.docx/.odt/.rtf` | V1 работает с текстом, не с вёрсткой; вывод — текст/markdown. |
| Повреждённый/нечитаемый файл | Понятное сообщение «не смог прочитать, пришлите в другом формате». |
| Длинные документы → лимит токенов / стоимость | Кап символов на запрос; при превышении — отказ с пояснением. |
| Качество русской редактуры у модели | Итерация промпта на эталонном наборе; опц. детерм. спелл-чек позже. |

## Эталонный набор (для этапов 3 и 5)
Собрать 5–10 реальных текстов агентства с известными «правильными» правками, **с «библией»/ТОВ
проекта** и **парами «исходник → правки клиента»**. Гонять после каждого изменения промпта.
Метрика: доля правок, принимаемых редактором без изменений.
