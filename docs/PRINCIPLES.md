# Принципы проекта

Этими принципами руководствуемся при любом решении. Они приоритетнее «как привычнее».

**Правило по умолчанию:** каждый компонент, сервис, технология, абстракция или паттерн
обязан обосновать своё существование. Если цель достигается более простым решением — выбираем его.

| Принцип | Что значит на практике |
|---|---|
| Requirements First | Сначала понять требование, потом решать. Не кодить вслепую. |
| Understand Before Modify | Прочитать и понять существующее до изменений. |
| Read More Than You Write | Больше читать (код, контекст), чем писать. |
| Fix Root Cause | Чинить причину, а не симптом. |
| KISS | Самое простое работающее решение. |
| YAGNI | Не строить то, что не нужно сейчас. |
| Avoid Premature Abstraction | Не абстрагировать раньше, чем появилось 2–3 реальных случая. |
| Boring Technology | Зрелые, скучные, проверенные инструменты вместо модных. |
| SOLID | Один компонент — одна ответственность; зависимости через интерфейсы. |
| DRY | Не дублировать логику и данные (правила/словарь — в одном месте). |
| Separation of Concerns | Слои разделены: адаптер ≠ извлечение ≠ ядро ≠ рендер. |
| Law of Demeter | Компоненты общаются через узкие интерфейсы, не лезут внутрь друг друга. |
| Optimize for Change | Стабильные швы (контракт `edit()`), чтобы менять части независимо. |
| Incremental Architecture | Растём по этапам; новое не ломает старое. |
| Convention over Innovation | Конвенциональные решения вместо изобретений. |
| Cost Awareness | Считать токены и инфраструктуру; не платить за лишнее. |
| Explicit Trade-Offs | Любой компромисс называть явно (плюс/минус). |

## Как это применено в V0
- Ядро `edit(text)` отделено от каналов — один контракт, любой канал как тонкий адаптер.
- Нет БД/очередей/Notion/агента/OCR на текущем скоупе — каждый пункт отложен осознанно.
- Контекст (правила, словарь) — данные в `config/`, а не код.
- Один вызов модели на все проверки; усложняем только по данным эталонного набора.
