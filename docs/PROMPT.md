# Промпт ядра редактуры

Самая нагруженная часть: от качества промпта зависит качество всего бота.
Промпт собирается в коде (`src/prompts.py`) из частей. Один вызов модели на текст.

## Структура промпта (один вызов)

1. **Системная инструкция** — роль и задача:
   > Ты — редактор русскоязычного текста. Исправь орфографию, пунктуацию и грамматику.
   > Найди неоправданные англицизмы и замени их русскими аналогами **прямо в тексте**.
   > Сохрани смысл и авторский голос; не переписывай без необходимости.
   > Верни результат строго в заданном JSON-формате.

2. **Правила стиля (данные)** — содержимое `config/rules.md`, подмешивается как есть.

3. **Словарь англицизмов (данные)** — из `config/anglicisms.json`:
   - `replace_always` — заменять всегда (термин → замена);
   - `context_dependent` — заменять с оглядкой на контекст (можно оставить, если уместно).

4. **Слот ТОВ (пока пустой)** — когда появятся библия/ТОВ, кладутся сюда же. Ядро не переписывается.

5. **Текст пользователя** — то, что прислали (после извлечения).

6. **Формат ответа** — требование вернуть **только JSON** по схеме ниже (без пояснений вокруг).
   Использовать структурный вывод модели (tool use / строгий JSON), чтобы надёжно разделить
   «исправленный текст» и «комментарии».

## Схема ответа (контракт `edit()`)
```json
{
  "edited_text": "готовый исправленный текст целиком",
  "summary": {
    "fixes_total": 0,
    "anglicisms_replaced": ["инсайт → вывод", "..."],
    "style_notes": ["короткое замечание по стилю", "..."]
  },
  "comments": [
    {
      "type": "ortho | punct | grammar | anglicism | style",
      "fragment": "исходный фрагмент",
      "change": "что стало",
      "reason": "коротко почему"
    }
  ]
}
```
`comments` — опционально (для разбора). Главный артефакт — `edited_text` + `summary`.

## Модель и стоимость
- Модель: `claude-sonnet-4-6` (баланс цена/качество для редактуры). Имя — из `.env` (`MODEL`).
- Стоимость линейна по длине текста; кап символов на запрос — `MAX_CHARS` в `.env`.
- Цены/лимиты модели — по докам Anthropic; не зашивать в код.

## Принципы для этого слоя
- Правила и словарь — **данные**, не хардкод (DRY, Optimize for Change).
- Один вызов на все проверки (KISS, Cost Awareness). Если качество одной проверки слабое —
  выносить её отдельно, и только тогда (YAGNI). Решение — по эталонному набору (см. PLAN.md).
- Держать правила компактными: длинный промпт = дороже и хуже фокус.
