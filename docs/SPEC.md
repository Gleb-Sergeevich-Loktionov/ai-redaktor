# Спецификация V0

Telegram-бот: на вход текст или файл → на выход отредактированный текст.
Версия 0.2. Связанные документы: `ARCHITECTURE.md`, `PLAN.md`, `PROMPT.md`, `PRINCIPLES.md`.

## 1. Скоуп и не-цели

**Цель V0:** дать креатору работающий инструмент за минимальную сборку. Бросаешь текст/файл
в Telegram — получаешь исправленный текст обратно. Один вызов модели, никакой инфраструктуры состояния.

### В скоупе
- Канал: Telegram-бот.
- Вход: вставленный текст и файлы `.txt`, `.md`, `.csv`, `.docx`, `.odt`, `.rtf`.
- Проверки: орфография, пунктуация, грамматика, англицизмы (поиск и замена неоправданных на русские аналоги).
- Лёгкая стилевая правка на основе общей нормы русского языка.
- Выход: файл с исправленным текстом + краткое саммари, что и почему поменялось.

### Сознательно НЕ в V0 (YAGNI — каждый отложенный пункт экономит сложность)
- **Полная редактура по ТОВ и стратегии** — ТЗ пока неполное, библии/ТОВ нет. Слот под них в промпте оставлен; подключаются, когда документы появятся.
- База данных, память между запросами, накопление контекста — это V2.
- Интеграция с Notion (запись результата в страницу) — отдельный адаптер позже.
- Агентная оркестрация — поток фиксирован, контроллер-ИИ не нужен.
- OCR и разбор PDF/сканов, распознавание сложной вёрстки и таблиц.
- Сохранение исходного форматирования/вёрстки в ответе — V0 возвращает текст/markdown (см. trade-offs).
- Аккаунты, роли, биллинг, веб-интерфейс, диф «было → стало» в UI.

## 2. Пользовательский сценарий
1. Креатор открывает бота, отправляет текст или файл (`.txt`, `.md`, `.csv`, `.docx`, `.odt`, `.rtf`).
2. Бот сразу отвечает «Обрабатываю…» (не блокирует, см. §5).
3. Через несколько секунд бот присылает: файл с исправленным текстом + сообщение с кратким разбором правок (сколько исправлений, какие англицизмы заменены, заметки по стилю).
4. Если файл не удаётся прочитать (повреждён, пустой, неподдерживаемый тип) или он слишком большой — бот понятно объясняет, что прислать.

## 3. Проверки (что делает ядро)
- **Корректура** — орфография, пунктуация, опечатки, согласование.
- **Грамматика** — управление, согласование, формы слов.
- **Англицизмы** — находит неоправданные, заменяет русскими аналогами прямо в тексте; опирается на словарь (`config/anglicisms.json`). Контекстно-допустимые не трогает.
- **Лёгкий стиль** — по общей норме и правилам из `config/rules.md`.
- **(Спящее) ТОВ/стратегия** — слот в промпте; включается, когда появятся документы.

## 4. Вход и выход

| Аспект | Решение V0 |
|---|---|
| Типы входа | Вставленный текст и `.txt`, `.md`, `.csv`, `.docx`, `.odt`, `.rtf`. `.pdf` — позже, через тот же интерфейс извлечения. |
| Извлечение | Плоские `.txt/.md/.csv` — чтением; форматированные `.docx/.odt/.rtf` — через **pandoc** → текст/markdown. |
| Лимит размера/длины | Кап на размер файла и число символов. При превышении — отказ с пояснением или разбиение на части (этап 4). |
| Долгая обработка | Бот сразу отвечает «Обрабатываю…», затем шлёт результат отдельным сообщением. Обработчик не блокируется. Очередь не нужна на текущем объёме. |
| Формат вывода | Файл с исправленным текстом (`.txt` / `.md`; `.docx` — опц.) + сообщение-саммари. Англицизмы заменены прямо в тексте. Исходная вёрстка не сохраняется. |
| Ошибки | Повреждённый/нечитаемый файл, неподдерживаемый тип, превышение лимита, пустое извлечение → понятные сообщения, что прислать. |

## 5. Trade-offs V0 (явные)

| Решение | Плюс / минус |
|---|---|
| Long-polling вместо webhook | + ноль инфраструктуры, нет публичного HTTPS. − хуже масштабируется; переключаемся на webhook при росте. |
| Один вызов модели на все проверки | + дёшево и просто. − если качество одной проверки слабое, выносим её отдельно. Решается измерением. |
| LLM-корректура без отдельного спелл-чека | + ноль доп. компонентов. − орфография у LLM не 100%. Митигация — опц. детерм. спелл-чек как дополнение. |
| Извлекаем текст, вёрстку не сохраняем | + единообразно для всех форматов, просто. − форматирование `.docx/.odt/.rtf` теряется; вывод — текст/markdown. Сохранение вёрстки — позже. |
| Вывод файлом, не дифом в UI | + ничего не строим. − нет наглядного «было → стало»; для V0 приемлемо. |

## 6. Definition of Done (приёмка V0)
- Бот принимает вставленный текст и файлы `.txt/.md/.csv/.docx/.odt/.rtf` и возвращает исправленный текст файлом + саммари.
- Орфография, пунктуация, грамматика и англицизмы обрабатываются на эталонном наборе реальных текстов так, что редактор принимает большинство правок.
- Неоправданные англицизмы заменены русскими аналогами прямо в тексте.
- Понятные сообщения на неподдерживаемый ввод (нечитаемый/повреждённый файл, слишком большой, пустое извлечение).
- Один деплой; бот стабильно онлайн; словарь/правила правятся без изменения кода.

## 7. Куда растём (швы сохранены)
- **ТОВ/стратегия:** кладутся в существующий слот промпта — ядро не переписывается.
- **Notion:** отдельный адаптер к тому же ядру `edit()` — Telegram остаётся.
- **Память (V2):** добавляется БД и слой памяти за тем же контрактом, без слома V0.
