# Спецификация V1 (MVP) — бот «redactor»

Telegram-бот: редактор отправляет текст или файл и получает отредактированный текст
+ перечень правок в контексте конкретного проекта (его «библии»/ТОВ).
Версия 1.0. Сменяет прежний скоуп V0 (без состояния): по решению заказчика «библия»
хранится по проектам в БД, добавлен поток клиентских правок.
Связанные документы: `ARCHITECTURE.md`, `PLAN.md`, `PROMPT.md`, `PRINCIPLES.md`.

## 1. Скоуп и не-цели

**Цель V1:** рабочий инструмент для редакторов агентства. В рамках проекта (со своей
«библией»/ТОВ) бот корректирует и редактирует текст, проверяет англицизмы и отдаёт
исправленный текст + перечень правок. Опционально сверяет текст с правками клиента.

### В скоупе
- Канал: Telegram-бот, long-polling.
- Проекты: создание/выбор проекта в чате; загрузка и обновление «библии»/ТОВ; хранение в БД.
- Вход: вставленный текст и файлы `.txt`, `.md`, `.csv`, `.docx`, `.odt`, `.rtf`.
- Проверки: орфография, пунктуация, грамматика, англицизмы; лёгкая стилевая правка по
  общей норме и по «библии»/ТОВ проекта.
- Клиентские правки (опционально): редактор подаёт исходный текст + версию с правками
  клиента, бот сверяет, применяет, комментирует.
- Выход: файл с исправленным текстом + краткое саммари + перечень правок
  (тип, фрагмент, что стало, почему).

### Сознательно НЕ в V1 (YAGNI — каждый отложенный пункт экономит сложность)
- **Память/накопление прошлых правок, обучение на предпочтениях** — это V2.
- **Notion-отгрузка** — отдельный адаптер позже (V2).
- **Детерминированный спелл-чек/морфология** (Hunspell/pymorphy) — полагаемся на LLM
  (явный компромисс, см. §7).
- OCR/PDF, сохранение исходной вёрстки `.docx/.odt/.rtf`, веб-UI, роли/биллинг, диф «было → стало» в UI.
- **Полноценный ACL:** в V1 «библию» проекта может править любой участник чата
  (внутренний инструмент; см. §7).
- Очереди, Kubernetes, мульти-инстанс.

## 2. Сущности и модель данных

- **Проект (Project)** — единица контекста. Поля: `id`, `chat_id`, `name`,
  `bible_text` (свободный текст: ТОВ, стратегия, термины, голос бренда), `updated_at`.
  «Библия» = `bible_text`, обновляется в любой момент командой.
- **Активный проект чата** — в каждом чате выбран один активный проект; команды редактуры
  работают в его контексте.
- **Глобальные данные** (файлы `config/`, не БД): правила стиля `rules.md`, словарь
  англицизмов `anglicisms.json` — общие для всех проектов. Проектная специфика терминов
  идёт в `bible_text`.
- **Истории правок НЕТ** — это V2. БД в V1 хранит только контекст проекта.

Схема БД — в `ARCHITECTURE.md`.

## 3. Пользовательские сценарии

### 3.1 Управление проектом
- `/newproject <имя>` — создать проект и сделать его активным.
- `/projects` — список проектов чата + какой активен.
- `/useproject <имя>` — переключить активный проект.
- `/setbible` — загрузить/заменить «библию» активного проекта (текст или файл следующим сообщением).
- `/showbible` — показать текущую «библию» активного проекта.

### 3.2 Редактура (основной поток)
1. Редактор отправляет текст или файл.
2. Бот сразу отвечает «Обрабатываю…» (не блокирует обработчик).
3. Бот берёт активный проект (`bible_text`) + глобальные правила/словарь, делает один вызов Claude.
4. Возвращает: файл с исправленным текстом + саммари + перечень правок.
5. Если активный проект не выбран — редактирует по общей норме и предупреждает, что контекст
   проекта не задан (мягкая деградация, не ошибка).

### 3.3 Клиентские правки (опционально)
- `/clientedits` запускает короткий диалог (`ConversationHandler`):
  шаг 1 — исходный текст/файл, шаг 2 — версия с правками клиента.
- Бот сверяет: применяет правки клиента, проверяет их на ошибки/англицизмы, отмечает
  расхождения с «библией», возвращает сведённый текст + перечень.

### 3.4 Ошибки
Нечитаемый/повреждённый/пустой файл, неподдерживаемый тип, превышение лимита размера/длины —
понятные сообщения, что прислать.

## 4. Проверки (что делает ядро)
- **Корректура** — орфография, пунктуация, опечатки, согласование.
- **Грамматика** — управление, согласование, формы слов.
- **Англицизмы** — находит неоправданные, заменяет русскими аналогами; опирается на
  `config/anglicisms.json`; контекстно-допустимые не трогает.
- **Стиль** — общая норма + `config/rules.md` + «библия»/ТОВ активного проекта.
- **Клиентские правки** — сверка, применение, комментирование (отдельная операция ядра).

## 5. Вход и выход

| Аспект | Решение V1 |
|---|---|
| Типы входа | Вставленный текст и `.txt/.md/.csv/.docx/.odt/.rtf`. `.pdf` — позже, через тот же интерфейс извлечения. |
| Извлечение | Плоские `.txt/.md/.csv` — чтением; `.docx/.odt/.rtf` — через **pandoc** в markdown. |
| Контекст | «Библия»/ТОВ активного проекта из БД подмешивается в промпт. Глобальные правила/словарь — из `config/`. |
| Лимиты | Кап на размер файла (`MAX_FILE_MB`) и число символов (`MAX_CHARS`). При превышении — отказ с пояснением. |
| Долгая обработка | Бот сразу отвечает «Обрабатываю…», затем шлёт результат отдельным сообщением. Обработчик не блокируется. |
| Выход — текст | Файл с исправленным текстом (`.md`). Англицизмы заменены прямо в тексте. Вёрстка не сохраняется. |
| Выход — перечень правок | **Обязателен.** Список правок (тип/фрагмент/что стало/почему) + саммари (счётчики, англицизмы, заметки по стилю). Короткий перечень — сообщением; длинный (> лимита Telegram) — отдельным файлом. |
| Ошибки | Повреждённый/нечитаемый/пустой файл, неподдерживаемый тип, превышение лимита → понятные сообщения. |

## 6. Хранение состояния

- **SQLite** — файл на диске/volume контейнера. Почему: per-project «библия» нужна по
  требованию заказчика; SQLite даёт ноль дополнительной инфраструктуры (Boring Technology,
  Cost Awareness). Без ORM (`sqlite3` из stdlib) — YAGNI.
- Репозиторий `store.py` изолирует БД от ядра и адаптера → переход на Postgres позже без
  правки ядра (Optimize for Change, DIP).
- Режим WAL; объём мал, конкуренция низкая, один инстанс.
- Это хранение **контекста проекта**, не история правок. Накопление/обучение — V2.

## 7. Trade-offs V1 (явные)

| Решение | Плюс / минус |
|---|---|
| БД (SQLite) в V1 вместо stateless | + per-project «библия» по требованию заказчика. − появляется состояние (схема, бэкап, команды). Митигация: SQLite + репозиторий-шов; хранится только `bible_text`, истории нет. |
| Long-polling вместо webhook | + ноль инфраструктуры, нет публичного HTTPS. − хуже масштабируется; webhook при росте. |
| Один вызов модели на все проверки | + дёшево и просто. − слабую проверку выносим отдельно по данным эталона. |
| LLM-корректура без спелл-чека | + ноль доп. компонентов. − орфография не 100%. Митигация — детерм. спелл-чек позже. |
| Перечень правок может быть большим | + ценно редактору. − лимит сообщения Telegram 4096 символов. Митигация: длинный перечень — файлом. |
| «Библию» правит любой в чате | + просто для внутреннего инструмента. − нет ACL. Митигация: ACL позже. |
| Извлекаем текст, вёрстку не сохраняем | + единообразно и просто. − форматирование теряется; вывод — текст/markdown. |

## 8. Definition of Done (приёмка V1)
- Проекты: создание/выбор/обновление «библии» работают; данные в БД переживают рестарт.
- Редактура учитывает «библию» активного проекта; отдаёт исправленный текст + саммари + **перечень правок**.
- Неоправданные англицизмы заменены в тексте; перечень показывает тип/фрагмент/правку/причину.
- Клиентские правки: поток работает, сводит версии и комментирует.
- Понятные сообщения на неподдерживаемый/повреждённый/пустой ввод и превышение лимита.
- Один деплой; БД на volume; глобальные правила/словарь правятся без изменения кода.

## 9. Куда растём (швы сохранены)
- **Память (V2):** история правок и предпочтений за тем же контрактом ядра (+ таблицы), без слома V1.
- **Notion (V2):** адаптер к тем же операциям ядра (`edit`/`apply_client_edits`); запись результата в страницу.
- **Per-project словарь/правила, ACL, спелл-чек:** швы оставлены (репозиторий, данные, контракт ядра).
